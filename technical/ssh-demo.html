## Understanding SSH in Relationships

SSH (Secure Shell) is how you connect to remote systems. In our metaphor, it's how you connect to another person's emotional core.

### Basic SSH Syntax

ssh [user]@[host]

# Example:
ssh unit-01@human.emotion.server

### Common Connection Failures

ssh: connect to host human.emotion.server port 22: Connection timed out
# Meaning: They're not ready to connect yet
# Solution: Wait. You cannot force this.

Permission denied (publickey)
# Meaning: You don't have their trust (the key)
# Solution: Earn it. You cannot steal it.

No route to host
# Meaning: They've blocked you completely
# Solution: Respect the boundary. Do not bypass it.

### The SSH Key Exchange

SSH uses public/private key pairs. This is the perfect metaphor for trust in relationships: 

# Generate your key pair
ssh-keygen -t ed25519 -C "your_email@example.com"

# This creates:
# - Private key (your vulnerability - keep it secret)
# - Public key (what you can safely share)

# To connect, you must:
# 1. Share your public key (show who you are)
# 2. They must add it to authorized_keys (grant access)
# 3. You connect using your private key (authentic vulnerability)

# You CANNOT:
# - Steal their private key (force intimacy)
# - Bypass authentication (fake trust)
# - Brute force the connection (pressure them)

### SSH Tunneling (The Deep Connection)

# SSH tunneling creates a secure channel
# Through which other connections can flow

ssh -L 8080:localhost:80 user@remote.server

# Metaphor:
# Once the SSH connection is established (trust),
# You can tunnel other protocols through it:
# - HTTP (daily communication)
# - Database connections (shared memories)
# - VNC (seeing their perspective)

# But the tunnel only works if the SSH connection is strong
# If trust breaks, all tunneled connections fail

### The Supervisor Connection (Root Access)

# Normal SSH connection
ssh user@server

# Escalating to root (granted, not stolen)
sudo su -

# This is Phase 5: The Supervisor
# Where they trust you with kernel-level access
# Not because you forced it
# But because you proved you'd protect, not control

### SSH Best Practices Applied to Love

1. Never share your private key
   (Keep your core self intact)

2. Use strong passphrases
   (Protect your vulnerability)

3. Respect failed connections
   (Accept "connection refused" gracefully)

4. Don't bypass authentication
   (Don't exploit vulnerabilities to force intimacy)

5. Monitor your authorized_keys
   (Know who has access to your heart)

6. Use connection timeouts
   (Some relationships require patience)

7. Keep logs
   (Remember the connection attempts, successful or not)
